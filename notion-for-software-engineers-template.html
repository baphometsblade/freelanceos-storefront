Architecture Decision Record

Why we chose Kafka over SQS

Context: needed ordered message delivery per partition and replay capability for audit trail. SQS doesn't guarantee order outside FIFO queues. Kafka gives us consumer group flexibility and retention-based replay. Trade-off: higher ops complexity, need MSK or self-managed cluster.

Decision: Kafka (MSK) for event streaming
Consequence: 2hr/wk ops overhead
Status: Accepted — 2025-11-14
Supersedes: SQS evaluation ADR-003
Debugging Post-Mortem

P0: API latency spike to 8s on checkout

Root cause: N+1 query in order summary endpoint. ORM generating 47 separate queries per request for related line items. Resolution: eager loading with JOIN, reduced to 2 queries. Detection: query logging in staging caught it before prod. Added query count assertion to integration tests.

Symptom: p99 latency 8200ms
Root cause: N+1 ORM query
Fix: .include('lineItems') + index
Prevention: query assertion in CI
Runbook

Postgres replica lag — diagnosis steps

Step 1: Check pg_stat_replication on primary. Step 2: Look at sent_lsn vs replay_lsn gap. Step 3: Check if replica IO is saturated (iostat). Step 4: If write throughput is high, check max_wal_size. Step 5: Consider adding replica or enabling parallel apply if PG 16+.

SELECT client_addr, state,
sent_lsn - replay_lsn AS lag_bytes
FROM pg_stat_replication;
System Design Note

URL Shortener — key design decisions

Requirements: 100M URLs, 10B reads/day, <10ms p99. Write path: hash + base62 encode → store in Redis + Postgres. Read path: Redis cache → Postgres fallback → 301 redirect. Consistent hashing for cache nodes. Bloom filter to avoid DB lookups for non-existent URLs.

Read:write ratio = 1000:1
Cache hit target: 99%+
Storage: ~100GB/year at 100B URLs
Key space: base62(7) = 3.5T combos
Technology Evaluation

Redis vs. Memcached — final comparison

Chose Redis for new projects due to: persistence (RDB + AOF), richer data structures (sorted sets for leaderboards, pub/sub for real-time, streams for event logs), Lua scripting for atomic operations, and Cluster mode for horizontal scale. Memcached wins only on raw throughput for simple k/v with multi-threading.

Winner: Redis for 99% of use cases
Exception: pure cache, max throughput
Benchmark: Redis ~100k ops/sec/node
Source: RedisConf 2025 + DDIA Ch6
API Design Note

Cursor-based pagination — implementation

Offset pagination breaks under concurrent inserts — rows shift, causing duplicates or skips. Cursor pagination uses an opaque token (usually base64-encoded last row ID + sort value). Client passes cursor param; server queries WHERE id > :cursor. Stable, consistent, and scales to billions of rows without performance degradation from large OFFSET values.

GET /api/posts?cursor=eyJpZCI6MTAwfQ
→ { data: [...], nextCursor: "..." }
Encode: btoa(JSON.stringify({id, ts}))
Index: (created_at DESC, id DESC)
