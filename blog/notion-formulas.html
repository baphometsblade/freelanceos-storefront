What Are Notion Formulas?

Notion formulas are expressions you add to any database as a special property type called "Formula." They let you compute new values from existing properties in the same database row — without leaving Notion or writing external code.

Think of them as the Notion equivalent of a spreadsheet formula, but wired directly into your database structure. Instead of referencing cells by address like A2 or B5, you reference properties by name using the prop("Property Name") function.

Notion formulas can perform a wide range of operations across every database in your workspace:

  • Perform arithmetic — add, subtract, and multiply revenue or hours fields
  • Compare values and output labels based on conditions (IF logic)
  • Manipulate dates — calculate deadlines, count elapsed days, add time windows
  • Transform text — concatenate names, strip characters, truncate long strings
  • Drive filtered views by outputting values that database filters can act on
  • Create dynamic status labels that update automatically as properties change

Every formula you create lives in a dedicated Formula column. The formula runs automatically for every row in the database — write it once and Notion keeps it current as the underlying properties change.

Formulas became significantly more powerful with the introduction of Notion Formula 2.0, which added support for richer functions, better date handling, and more consistent type behavior. All 20 examples in this guide are written for Formula 2.0 syntax, which is the current standard for all Notion workspaces.

The real power of Notion formulas emerges when you combine them with database views. A formula that outputs "Overdue" or "High Priority" becomes genuinely useful when paired with a filtered view that shows only rows matching that label — giving you a live, self-maintaining task list with zero manual curation.

Tip: You can filter, sort, and group database views by any Formula property — making formulas one of the most powerful organizational tools in your entire Notion workspace.

Formula Syntax and Property Types

Before writing your first formula, you need to understand how Notion handles data types. Every property in your database has a type — text, number, checkbox, date, select, etc. — and formulas enforce strict type matching. Passing the wrong type into a function is the most common source of formula errors.

Core Syntax Rules

  • Reference a property: prop("Property Name") — the name must match exactly, including capitalization and spaces.
  • String values: wrap in double quotes — "Done", "High", "Active"
  • Booleans: use true or false in lowercase, no quotes
  • Arithmetic operators: +, -, *, / work on numbers as expected
  • Comparison operators: == (equal), != (not equal), >, <, >=, <=
  • Nesting: functions can nest inside each other — if(and(cond1, cond2), result, fallback)
  • String concatenation: use + between strings or use the concat() function

Data Type Compatibility

Notion formulas enforce strict type rules. Comparing a number property to a text string will throw an error. Here are the key type behaviors to know:

  • Select and multi-select properties are treated as text in formulas — compare them with == "Option Name"
  • Checkbox properties are booleans — compare with == true or use them directly as a condition
  • Date properties require date-specific functions like dateBetween() and dateAdd()
  • Number properties support standard arithmetic operators directly
  • To mix types in output: wrap numbers in format() to convert them to text for concatenation

How to Add a Formula Property

In any Notion database, click the "+" button at the end of the property row (or at the top of any column in table view). Choose "Formula" as the property type. The formula editor opens, with autocomplete for property names and functions. Write your expression and press Enter to save.

Note: Notion Formula 2.0 removed many implicit type coercions that existed in earlier versions. If you are migrating old formulas that used to work, you may need to add explicit format() or toNumber() calls where types previously converted automatically.

IF, AND, OR Logic Formulas (Formulas 1–5)

Conditional logic is the backbone of useful Notion formulas. The if() function lets you output different values depending on whether a condition evaluates to true or false. Combine it with and() and or() to handle scenarios where multiple conditions must be checked simultaneously.

The general pattern for IF logic is:

if(condition, value_when_true, value_when_false)

For nested conditions, place another if() in the false position:

if(condition1, result1, if(condition2, result2, default_result))
Formula 1 — Checkbox From Status
Returns a boolean (true/false) based on whether the Status select field equals "Done". Useful for driving filtered views that show only incomplete items, and for rollup counts of completed records.
if(prop("Status") == "Done", true, false)
Formula 2 — Client Tier Label
Uses AND logic to classify clients. Only labels a client "High Value" when both conditions are true: revenue above $1,000 AND status is Active. If either condition fails, the client gets "Standard". AND requires every condition to be true.
if(and(prop("Revenue") > 1000, prop("Status") == "Active"), "High Value", "Standard")
Formula 3 — Urgency Flag
Uses OR logic to flag urgent items. A task gets marked urgent if it is either high priority OR already overdue — only one condition needs to be true. Outputs an emoji label suitable for board and gallery views.
if(or(prop("Priority") == "High", prop("Overdue") == true), "🔴 Urgent", "Normal")
Formula 4 — Deadline Check
Uses the empty() function to detect whether a date field has been filled in. Perfect for filtering views to surface tasks that are missing a due date, so nothing slips through without a deadline assigned.
if(empty(prop("Due Date")), "No deadline", "Has deadline")
Formula 5 — Budget Alert
Compares two numeric properties to catch projects where actual hours have met or exceeded the estimate. The result updates live as you log hours, giving you a real-time budget health indicator per project.
if(prop("Hours Logged") >= prop("Hours Estimated"), "⚠️ Over budget", "On track")

You can nest IF statements for multi-step classification. Notion supports deep nesting — though formulas with more than three or four levels become difficult to maintain. For complex logic, consider breaking the work into intermediate formula columns that each handle one step.

Date Formulas: dateAdd, dateBetween, now() (Formulas 6–10)

Date handling is one of the most requested uses of Notion formulas. Whether you are tracking deadlines, measuring project age, setting automatic review windows, or flagging overdue work, these five formulas cover the most common date calculation patterns used by freelancers and teams.

The key date functions available in Notion formulas:

  • now() — returns the current date and time, updated dynamically every time the database loads
  • dateAdd(date, number, unit) — adds time to a date; units include "days", "weeks", "months", "years", "hours", "minutes"
  • dateBetween(date1, date2, unit) — returns the signed difference between two dates in the given unit
  • format(date) — converts a date value to a human-readable text string
  • dateStart(date) and dateEnd(date) — extract the start or end of a date range property
Formula 6 — Add 7 Days to Start Date
Calculates a check-in or review date 7 days after the project start. Change the number and unit to suit any interval — 2 weeks, 1 month, 90 days. The result is a date property usable in views and filters.
dateAdd(prop("Start Date"), 7, "days")
Formula 7 — Days Until Due
Returns the number of days remaining until the due date. A positive value means days remaining; a negative value means the item is overdue by that many days. Perfect for a live countdown column in any project tracker.
dateBetween(prop("Due Date"), now(), "days")
Formula 8 — Age of Record in Days
Calculates how many days have passed since the record was created. Useful for lead tracking, support ticket aging, or surfacing projects that have been open without updates for too long.
dateBetween(now(), prop("Created"), "days")
Formula 9 — 30 Days From Today (Text Output)
Wrapping dateAdd in format() converts the computed date to a human-readable text string rather than a date value. Useful when you want to display a computed date inside a label or concatenated with other text.
format(dateAdd(now(), 30, "days"))
Formula 10 — Overdue Flag
Combines dateBetween with an IF to label tasks as Overdue when the days-until-due drops below zero. Use this as a filter in a dedicated "Overdue" database view that needs zero manual maintenance to stay accurate.
if(dateBetween(prop("Due Date"), now(), "days") < 0, "Overdue", "On time")

Important: now() recalculates every time someone opens or refreshes the database — it is not a snapshot. If you need a fixed timestamp for when something was created or completed, use the built-in "Created time" or a dedicated date property set manually at the time of the event.

Text Functions in Notion (Formulas 11–13)

Notion's text functions let you transform, combine, and extract information from text properties. These are particularly useful for databases that hold names, URLs, or long descriptions that need to be standardized or shortened for display in gallery views, board cards, or notification labels.

Formula 11 — Full Name From First and Last
Concatenates a First Name and Last Name property with a space in between. The concat() function accepts any number of arguments and joins them in sequence — add middle names, titles, or suffixes by including more arguments.
concat(prop("First Name"), " ", prop("Last Name"))
Formula 12 — Uppercase Name
Converts the Name property to all uppercase characters. Pairs naturally with lower() (all lowercase) and title() (Title Case) for normalizing inconsistently entered data imported from external sources or forms.
upper(prop("Name"))
Formula 13 — Truncate Notes to 100 Characters
The slice(text, start, end) function extracts a substring starting at the given index. Starting at 0 and ending at 100 gives the first 100 characters — ideal for previewing long notes or descriptions in gallery and board card layouts.
slice(prop("Notes"), 0, 100)

Additional text functions worth knowing:

  • contains(text, substring) — returns true if text includes the substring; useful in IF conditions
  • replace(text, search, replacement) — replaces only the first match found
  • replaceAll(text, search, replacement) — replaces every occurrence of the search string
  • test(text, regex) — tests whether text matches a regular expression pattern
  • trim(text) — removes leading and trailing whitespace
  • split(text, delimiter) — splits text into a list; useful in Formula 2.0 for processing comma-separated values

Number and Math Formulas (Formulas 14–16)

Number formulas handle arithmetic, rounding, and financial calculations. They are the workhorses of Notion expense trackers and income dashboards — anywhere you are doing math on numeric properties to produce summaries, estimates, or alerts.

Formula 14 — Character Count of Description
Returns the number of characters in a text property as a number. Useful for enforcing character limits on meta descriptions, Twitter bios, email subject lines, or any field where length matters for the downstream context.
length(prop("Description"))
Formula 15 — Strip HTTPS From URL
replaceAll() finds every occurrence of "https://" in the URL field and replaces it with an empty string, leaving just the domain and path. Useful for cleaner display in gallery views and for normalizing URLs before using them in other formulas.
replaceAll(prop("URL"), "https://", "")
Formula 16 — 30% Tax Estimate
Multiplies revenue by 0.3 and rounds to the nearest integer. A practical self-employment tax reserve calculator. Adjust the multiplier to match your actual effective tax rate — 0.25, 0.35, or whatever your bracket requires.
round(prop("Revenue") * 0.3)

Additional math functions available in Notion formulas:

  • abs(number) — absolute value (removes the negative sign)
  • ceil(number) — round up to the nearest integer
  • floor(number) — round down to the nearest integer
  • max(a, b) and min(a, b) — return the larger or smaller of two values
  • mod(a, b) — remainder after division
  • pow(base, exponent) — raise a number to a power
  • sqrt(number) — square root
  • log(number) — natural logarithm

Checkbox and Conditional Formulas (Formulas 17–18)

Two of the most universally useful formulas in freelance and business databases involve tracking project value and net profit. These formulas are the foundation of any serious Notion for business operations setup — they produce live financial numbers that feed dashboards, rollups, and client health scores.

Formula 17 — Total Project Value
Multiplies a Rate (hourly rate) property by a Hours property to calculate the total billable amount for a project. This updates live as you adjust either number, giving you a running project value without any manual calculation.
prop("Rate") * prop("Hours")
Formula 18 — Net Profit
Subtracts Expenses from Revenue to return net profit per row. When combined with a Rollup that sums this formula column across all projects for a given client, you get a real-time profitability score per client relationship.
prop("Revenue") - prop("Expenses")

Pro move: Create a Formula property for net profit per project, then add a Rollup on a Client database that sums those net profit values via a Relation. You get a live total profitability number for every client, automatically updated as you add or modify project records.

Rollup + Formula Combinations (Formulas 19–20)

The most sophisticated Notion setups combine Rollup properties with Formula properties. A Rollup brings aggregate data in from a related database through a Relation link. A Formula then performs further logic on those aggregated values. The result is multi-database intelligence surfaced in a single row — giving you a bird's-eye view of complex relationships without leaving the database.

Before using these formulas, you need a Relation property linking your database to another (e.g., Projects linked to a Tasks database). Add a Rollup that counts or sums a field from the related records. Then your formula can reference the Rollup as if it were a regular numeric property.

Formula 19 — Progress Percentage
If Tasks Done equals Total Tasks, shows a completion indicator. Otherwise, calculates the percentage complete and displays it as text. Both Tasks Done and Total Tasks are typically Rollup properties counting related task records in a linked Tasks database.
if(prop("Tasks Done") == prop("Total Tasks"), "✅ Complete", format(round(prop("Tasks Done") / prop("Total Tasks") * 100)) + "% done")
Formula 20 — Client Rating Tier
Uses nested IF statements to classify clients into three tiers based on a Rating Rollup — for example, the average star rating pulled from linked project feedback records. Top clients get highlighted; at-risk clients surface for proactive outreach.
if(prop("Rating") >= 4, "⭐ Top Client", if(prop("Rating") >= 2, "Regular", "At Risk"))

To implement these rollup-formula combos in practice, follow this three-step setup:

  1. Create a Relation from your primary database (e.g., Clients) to your secondary database (e.g., Projects)
  2. Add a Rollup on the primary database to count related projects, sum revenue, or average a score from the linked records
  3. Add a Formula property that references the Rollup to compute a tier, health score, or progress label

This three-layer pattern — Relation, Rollup, Formula — is the foundation of every serious Notion CRM, project tracker, and financial dashboard. Master it and you can build systems that rival dedicated software tools.

Real-World Formula Examples

Let's look at four complete, real-world formula implementations that freelancers and solopreneurs use every day. These show how individual formulas combine with database structure to create genuinely useful systems.

Days Until Deadline (Project Tracker)

In a project tracker database with a "Due Date" property, add a Formula column called "Days Left" using Formula 7. Then create a filtered view called "Due This Week" that filters where Days Left is between 0 and 7. Sort by Days Left ascending. You now have a live "due this week" view that never needs manual updating — it recalculates every time you open it.

dateBetween(prop("Due Date"), now(), "days")

Automated Project Status

Combine a "Delivered" checkbox property with the progress formula and a "Not Started" guard for empty projects. This single formula replaces a manual status select that people forget to update:

if(prop("Delivered") == true, "✅ Delivered", if(prop("Total Tasks") == 0, "Not Started", if(prop("Tasks Done") == prop("Total Tasks"), "Ready to Deliver", format(round(prop("Tasks Done") / prop("Total Tasks") * 100)) + "% done")))

Income Tracker (Paid vs. Unpaid)

An income tracker database with "Invoice Date", "Amount", and "Paid" (checkbox) properties. Add a Formula column for received amount that outputs the invoice value only when paid — otherwise zero. A Rollup that sums this column gives you total cash actually received, not just invoiced:

if(prop("Paid") == true, prop("Amount"), 0)

Retainer Hours Remaining

For retainer-based client relationships where each client has a monthly hours allocation, track usage with a Rollup sum of logged hours and calculate what remains. This formula also flags overages clearly:

if(prop("Hours Used") > prop("Monthly Hours"), "⚠️ " + format(prop("Hours Used") - prop("Monthly Hours")) + "h over", format(prop("Monthly Hours") - prop("Hours Used")) + "h remaining")

Common Notion Formula Errors and How to Fix Them

Notion displays a red warning icon when a formula has an error. The editor often provides a hint about what went wrong, but the root cause is not always obvious. Here are the most common issues and how to resolve each one.

Type Mismatch Error

This happens when you pass a value of one type where another is expected — most commonly when trying to concatenate a number into a text string without converting it first.

Fix: wrap numeric values in format() before using them in string operations.

// Incorrect - will throw a type error: "Revenue: " + prop("Revenue") // Correct - format() converts the number to text first: "Revenue: " + format(prop("Revenue"))

Property Not Found

Notion throws this error when the property name in your prop() call doesn't match any column in the database. This is almost always a spelling or capitalization mistake. Property names are case-sensitive.

Fix: click into the formula editor and use the autocomplete dropdown. It lists all available properties so you can select the exact name rather than typing it manually. This eliminates the entire category of "property not found" errors.

Division by Zero

If you divide by a property that could be zero — like Total Tasks when a project hasn't started yet — Notion returns an error for that row. This doesn't break the formula for other rows, but it creates a messy-looking column.

Fix: guard against zero with an IF check before the division:

if(prop("Total Tasks") == 0, "Not started", format(round(prop("Tasks Done") / prop("Total Tasks") * 100)) + "%")

Empty Property in Date Formula

Date functions throw errors when a date property is empty — which happens often if not every record has been given a due date yet. Always wrap date formulas in an empty() check to handle the unfilled case gracefully:

if(empty(prop("Due Date")), "No date set", format(dateBetween(prop("Due Date"), now(), "days")) + " days")

Using Single Equals for Comparison

In Notion formulas, equality comparison uses == (double equals), not = (single equals). A single equals sign inside a formula body causes a syntax error. Similarly, inequality uses !=, not <>.

Formula Works on Some Rows, Errors on Others

This usually means some rows have empty or unexpected values in the properties your formula references. Add empty() guards for every property that might be unfilled. Use the formula editor's preview to test against specific rows in your database before saving.

Tips for Writing Better Formulas

After working with Notion formulas across dozens of real-world databases, these practices make a consistent difference in how reliable, readable, and maintainable your formulas turn out to be.

Use the Autocomplete for Every Property Reference

The Notion formula editor autocompletes property names as you type prop(". Always select from the dropdown rather than typing property names by hand. This single habit eliminates the most common class of formula errors entirely — misspelled or wrongly capitalized property names.

Break Complex Logic Into Layers

If a formula is getting hard to read or debug, split it into intermediate Formula properties. For example: create a "Days Overdue" formula first, then reference that formula property in a separate "Status Label" formula. This is dramatically easier to maintain than one deeply nested 200-character expression, and lets you debug each step independently.

Always Guard Against Empty Values

Date properties and Relation-derived values can be empty. Use empty(prop("Property Name")) as a guard at the beginning of any formula that depends on a field that might not always be filled in. Write the guard first, test with an empty row, then build the rest of the formula.

Test With Deliberate Edge Cases

Create a test row with extreme or unusual values — zero hours, a past due date, an empty name field, maximum possible numbers — and confirm your formulas handle them gracefully before applying the database to real work. Catching edge cases in testing is far less painful than finding them live during a client review.

Name Your Formula Columns to Describe the Output

Use column names that describe what the formula produces, not what it calculates. "Days Until Due" is better than "dateBetween formula". "Project Status" is better than "IF status formula". Good names make filtered views self-explanatory to anyone else who accesses the database — including future you.

Pair Every Formula With a View

A formula that outputs "Overdue" only delivers value when there is a database view filtered to show rows where that formula equals "Overdue". Build the view at the same time you build the formula. The combination of formula + view is what transforms Notion from a manual database into a self-organizing system.

Document Non-Obvious Formulas

For any formula that requires explanation, add a "Formula Notes" text property to the database where you describe what the formula does, which properties it depends on, and what valid output values look like. This pays off enormously when returning to a complex database after weeks away from it.

FAQ

Why are my Notion formulas showing errors?

Notion formula errors most commonly come from one of four sources: a type mismatch where you are passing text where a number is expected (or vice versa), referencing a property name that does not exist or is misspelled (remember names are case-sensitive), unbalanced or missing parentheses, or division by zero when a numeric property is empty or zero. Start by checking that your prop() names exactly match the database column names using the autocomplete. Then verify you are comparing matching data types. Add empty() guards for any property that might be unfilled. Most errors fall into one of these categories.

Can Notion formulas reference other databases?

Notion formulas cannot directly reference properties from other databases. However, the workaround is effective and widely used: create a Relation property to link two databases, then add a Rollup property that aggregates a specific field from the related records (sum, count, average, etc.). Once that Rollup property exists in your current database, your formula can reference it using prop("Rollup Name") exactly like any other property. This Relation + Rollup + Formula three-layer pattern unlocks genuine cross-database calculations.

How do I use IF statements in Notion?

The Notion IF formula uses the syntax if(condition, value_if_true, value_if_false). A basic example: if(prop("Status") == "Done", true, false). For multi-tier logic, nest IF statements by placing a second IF in the false position: if(condition1, result1, if(condition2, result2, default_result)). You can use and() and or() inside the condition to test multiple criteria simultaneously: if(and(condition1, condition2), "Both true", "Not both").

What is the difference between a rollup and a formula in Notion?

A Rollup pulls aggregated data (sum, count, average, min, max, and more) from a related database through a Relation property. It crosses database boundaries to bring in information from linked records. A Formula performs calculations or transformations entirely on properties within the same database row — it can only see data from that single entry. The two work together powerfully: a Rollup brings in external data from linked records, and a Formula then uses that rollup value in further calculations, comparisons, or label generation to produce the final output you need.

Can I use formulas in Notion database filters?

Yes, absolutely. Once you create a Formula property in a database, it becomes available as a filter criterion in any view of that database, just like any other property type. For example, if your formula outputs "Overdue" for certain rows, you can create a filtered view that only shows rows where the Formula property equals "Overdue". This is one of the highest-value uses of Notion formulas — computed values driving intelligent, always-current filtered views with zero manual curation.

How do I calculate the number of days between two dates in Notion?

Use the dateBetween() function. The full syntax is dateBetween(date1, date2, unit) where unit is a quoted string: "days", "weeks", "months", "years", "hours", or "minutes". To find days until a due date: dateBetween(prop("Due Date"), now(), "days"). To find how old a record is: dateBetween(now(), prop("Created"), "days"). The result is a positive or negative number — negative means the first date is earlier than the second, which is how you detect overdue items.

Can Notion formulas do conditional formatting?

Notion does not have native cell-level conditional formatting like Excel or Google Sheets, where individual cells change background color based on their value. However, you can simulate visual indicators by having formulas output different emoji or labeled text ("🔴 Urgent" vs. "Normal") and then surfacing those labels prominently in gallery card layouts, board view column groupings, or filtered views. Notion's database row-coloring feature can apply colors to entire rows based on select field values, which pairs well with formulas that write to a select-like text column.

Are Notion formulas the same as Excel formulas?

They share conceptual inspiration but differ in important ways. In Notion, formulas reference the current row's properties by name using prop('Property Name') rather than cell addresses like A1 or B2. Notion formulas are isolated to a single row — they cannot reference other rows, aggregate across the database, or look up values from the same column. For aggregation across rows, you use Rollup properties instead. The available functions are a focused subset of what Excel offers, covering text manipulation, date arithmetic, conditional logic, and basic math. Excel is more powerful for pure numerical analysis; Notion is purpose-built for structured relational data management where formula-driven automation handles the rest.
