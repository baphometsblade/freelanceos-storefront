The Repository

Building an Atomic Research Repository

The most valuable thing in this template isn't a single module — it's the way the Insights Repository connects every piece of evidence to every decision. Here's how the architecture works.

The Atomic Insights Model

An atomic insight is a single, discrete observation: one behavior, one belief, one pain point, sourced to specific evidence. Instead of a study report that says "users found navigation confusing," you store twenty atomic observations about navigation, each linked to the session where it was observed. When you filter the repository by "navigation," you see all twenty observations across every study you've ever run.

  • One claim per insight. Not "users found X and Y confusing and also mentioned Z." Three separate insights, each with its own evidence link.
  • Confidence levels. Single observation vs. pattern across 5+ sessions vs. quantitatively validated. Stakeholders can filter by confidence before making decisions.
  • Insight aging. A property tracks how old each insight is. Insights over 18 months surface a "Verify?" flag — prompting you to check whether the finding still holds after product changes.
  • Evidence linking. Every insight links to the specific session record (or records) that support it. One click from an insight to the raw note. One click from a persona to the insights that define it.

Tagging Taxonomy

A consistent tag system is what makes the repository searchable years from now. The template ships with a two-layer taxonomy:

Theme tags (what the insight is about):

navigation onboarding trust pricing checkout search notifications mobile error-handling performance

Context tags (where/who):

new-user power-user enterprise mobile-only Q1-2025 discovery

Surfacing Insights in Design Reviews

The repository is only useful if it gets used during design work. The template includes three views designed for design review sessions:

  • By product area. Filter all insights related to the feature being designed. Skip the literature review — the relevant findings are already tagged.
  • High-confidence only. Filter to insights with 5+ supporting observations. Lead with the strongest evidence in design critiques.
  • Recent vs. foundational. Separate insights from the last 6 months (current behavior) from older foundational findings (persistent mental models).
  • Decision links. After a design decision is made, link it back to the insights that supported it. Creates a traceable rationale for future teams.
Interview & Usability Testing

From Session Prep to Processed Findings

Good research output starts before the session opens. The template covers the full arc from discussion guide to affinity map.

Interview Preparation

  • Discussion guide template. Research question → interview questions → probes. Structured so the guide is scannable during a live session without reading off a script.
  • Consent form template. Covers recording consent, data use, anonymization, and right to withdraw. Stored as a linked page in each session record.
  • Screener template. Demographic and behavioral screener questions stored in the participant record, not a separate spreadsheet.
  • Participant tracker. Recruitment status, confirmation, show/no-show, and incentive payment — all tracked in the interview database.

Note-Taking Structure

  • Consistent heading structure. Every session note follows: Background recap / Task walkthrough / Key moments / Direct quotes / Researcher observations. Consistent structure makes cross-session analysis faster.
  • Quote capture. A dedicated quotes block in each session record for verbatim captures. Quotes link to insights and personas.
  • Timestamp anchors. If you record sessions, timestamp fields let you link directly to the moment in the recording that supports an insight.
  • Observer notes. Separate section for co-researcher or observer notes, keeping the primary note-taker's thread clean.

Affinity Mapping in Notion

Notion is not Miro. But it does something Miro doesn't: every sticky-note equivalent stays linked to the source session. The template's affinity mapping approach uses a board view of the Insights Repository, grouped by theme tag. You work through sessions, pulling atomic insights out of notes and into the repository. As you tag them, the board view clusters them automatically. The result is an affinity map where every cluster element links to its evidence — something you can't get from a photo of a wall of stickies.

  • Board view by theme. Drag insights between theme columns as patterns emerge. Rename and merge themes as the analysis develops.
  • Frequency view. Sort by number of supporting observations to identify the patterns with the strongest evidence base.
  • Sub-theme nesting. Use linked databases to nest sub-themes under primary themes for complex study synthesis.

Usability Test Findings: Severity Ratings

The Usability Test Logger uses a four-level severity taxonomy derived from Nielsen's heuristic evaluation model, adapted for moderated and unmoderated testing:

  1. Critical Task failure or abandonment. Must be fixed before launch. Linked to blocking design recommendations.
  2. Serious Significant friction or delay. Should be fixed before launch. Linked to recommended design improvements.
  3. Minor Confusion or mild friction. Address when possible, not a blocker.
  4. Cosmetic Polish issue with no impact on task completion. Capture for backlog, not a priority.
For Freelance UX Researchers

Managing Multiple Clients Without Losing Your Mind

Freelance UX researchers face a challenge in-house researchers don't: context-switching between clients, keeping work siloed, and billing accurately. The template is built for both scenarios.

Multi-Client Project Management

  • Client workspace isolation. Each client gets a scoped database view so their research never surfaces in another client's filters. All data lives in the same system — what changes is the lens you look through.
  • Project-level access control. Share client-specific sub-pages rather than the full workspace. Clients see their research, not yours or anyone else's.
  • Cross-client insight patterns. When you're not sharing with clients, you can query across all projects to see if patterns from Client A's research might be relevant to Client B's problem space.
  • Research request intake per client. Each client has their own request queue view. Requests come in, you triage, and nothing falls through the cracks.

Deliverable Templates

  • Research brief. Pre-study document covering objectives, questions, methodology, timeline, and success criteria. Share with clients before kickoff.
  • Topline report. One-page summary of key findings for immediate-turnaround stakeholder needs.
  • Full research report. Structured template covering methodology, participant profile, findings by theme, recommendations, and next steps.
  • Insight summary card. Single-page visual of the top 5–7 findings with supporting evidence. Design-team and executive friendly.

Billing & Project Tracking

  • Time tracking per study. Log hours by phase: planning, recruitment, facilitation, analysis, reporting. Know your actual time-per-deliverable.
  • Project budget view. Estimated vs. actual hours per project. Surfaces scope creep before it becomes a billing conversation problem.
  • Invoice trigger dates. Milestone-based invoicing prompts linked to study phase completions.
  • Client-facing vs. internal notes. Clear separation between raw working notes (internal only) and polished deliverables (client-facing). Never accidentally share the wrong thing.

The client-facing/internal split matters more than most freelancers realize. Your raw session notes contain unfiltered observations, shorthand, and working hypotheses that can be misread without context. The template enforces this separation structurally — internal pages are never in the client-facing database views. You can share a research brief, an insight summary, or a full report without worrying about exposing your working notes.

Stakeholder Communication

Research That Gets Read, Not Archived

The best research in the world doesn't move products if it lives in a report no one opens. The template includes four formats designed for how stakeholders actually consume research.

  1. 1

    Insight Summary (1-pager)

    Five to seven key findings from a study, each with a confidence indicator and a one-sentence implication. Designed to be read in a 5-minute Slack scan. Links to the full report for anyone who wants more. This is the format stakeholders actually share with each other.

  2. 2

    Executive Brief

    The "so what" layer above the insight summary. One paragraph of context, three findings with business implications, and one recommended action. Designed for senior stakeholders who need to make a go/no-go decision, not a design critique.

  3. 3

    Evidence Gallery

    A curated collection of direct quotes, video clips, and screenshots that make the research concrete. Stakeholders who need to feel the research — not just read it — can browse the gallery. Particularly effective for convincing skeptics who dismiss summary findings as subjective.

  4. 4

    Living Document vs. Frozen Report

    The template distinguishes between living documents (the Insights Repository, personas — updated as new research comes in) and frozen reports (the deliverable from a specific study — timestamped and unchanging). Stakeholders know the difference: a living document gives you the current state of knowledge; a frozen report gives you what was true at a specific point. Both have their place.

The Stakeholder Research Request Queue closes the loop: after a study delivers, the original request record gets updated with a link to the deliverable. Whoever asked the research question gets notified and can trace the full journey from question to finding to recommendation.

FAQ

Common Questions

Everything researchers ask before switching to a Notion-based research system.

Can I replace Dovetail with Notion for UX research?
For many solo researchers and small teams, yes. Dovetail excels at auto-tagging transcripts at scale, but its $30–$100/month per-seat cost is hard to justify for independent researchers. FreelanceOS Pro's Notion research system covers the core workflow: storing interview notes, tagging atomic insights, linking evidence to findings, building personas, and sharing results with stakeholders. If you're running 2–5 studies per quarter rather than 20, Notion gives you 80% of Dovetail's value at a fraction of the cost. The gap closes further if you use a separate AI transcription tool (Otter, Fireflies, or Grain) and import summaries into Notion.
How do I tag research insights across multiple projects?
The template uses a two-layer tag taxonomy: theme tags (navigation, onboarding, trust, pricing, etc.) and project/client tags. Every atomic insight in the Insights Repository carries both layers, so you can filter by theme across all projects or drill into a single study. A dedicated Tags reference database lets you define and standardize tag meanings so the taxonomy stays consistent as your research library grows. The most important rule: define your theme taxonomy before you start tagging, not after. The template ships with a starter taxonomy you can adapt for your product domain.
Can my design team access my Notion research repository?
Yes. The template separates internal research workspaces from shareable views. You can give designers read-only access to the Insights Repository and Persona Hub without exposing raw session notes or consent forms. Stakeholder-facing Insight Summaries are designed as standalone pages that stand alone without context — no training required to read them. For teams on Notion's Team plan, you can set up granular page-level permissions so each audience sees exactly what they need.
How do I track the impact of my research on product decisions?
The Research Roadmap module includes an Impact Tracking field on every research study. After a study closes, you link product decisions, design changes, or backlog items that were influenced by its findings. Over time this builds a paper trail showing exactly how research shaped the product — useful for performance reviews, stakeholder trust, and proving research ROI. The template includes an Impact Summary view that aggregates all research-to-decision links in a single dashboard, filterable by quarter.
What is the best way to organize interview notes in Notion?
The template uses a two-layer approach: each interview session is a database record with structured metadata (participant, date, study, screener answers, consent status), and the notes live inside that record using a consistent heading structure (Background, Task walkthrough, Key moments, Direct quotes, Researcher observations). After each session you pull atomic insights out of the notes and add them to the Insights Repository, keeping raw notes separate from processed findings. The consistent structure matters more than the exact headings — whatever you use, use it every time so cross-session scanning becomes fast and reliable.
Can I run surveys from Notion?
Notion doesn't collect survey responses natively, but the Survey & Quantitative Tracker in this template is built around that reality. You run surveys in Typeform, Google Forms, or Maze, then import summary data and key verbatims into Notion. The tracker stores response counts, completion rates, top findings, and links to raw data exports — keeping quantitative and qualitative data in the same repository without trying to force Notion into a survey tool. For simple screeners and feedback forms, Notion forms (available on paid plans) can handle basic collection.
